🏗️ Arquitectura actual

Backend en PHP plano con API REST modular (api/modules/*) y consultas preparadas (PDO). Frontend SPA en JavaScript vanilla (ES6 modules), sin frameworks. Tests E2E con Playwright.

  • Dashboard — KPIs generales del periodo
  • Periodos — control de ciclos mensuales (ABIERTO / CERRADO / ANULADO)
  • Inquilinos, Unidades, Ocupaciones — propiedades, inquilinos y contratos
  • Recibo de luz, Lecturas, Liquidación — consumo eléctrico y prorrateo con IGV
  • Cobros, Avisos, Tarifas — facturación mensual y avisos de pago

✅ Lo que ya funciona bien

  • Separación por conceptos en cobros_mensuales (alquiler, luz, agua, gas, otros, mora, descuento)
  • Cálculo de consumo eléctrico por medidor con prorrateo de gastos comunes e IGV (18%)
  • Control de periodos que evita duplicar o solapar ciclos mensuales
  • Avisos visuales (PNG) listos para compartir por WhatsApp/Yape, con deuda anterior
  • Consultas parametrizadas (PDO) en todo el código — buena protección contra SQL injection

Sin autenticación ni roles Toda la API está abierta: no hay login, sesiones ni usuarios.
Alto

Alto

Medio
Sin notificaciones automáticas Los avisos se generan, pero el envío es 100% manual (copiar y compartir).
Medio
Sin recibos PDF Solo se genera PNG vía canvas; no hay comprobantes imprimibles/oficiales.
Medio

Bajo
Archivos sueltos en la raíz del proyecto insertar_enero.php, resumen_enero.php, plantillas TSV/TXT y prueba.html parecen scripts de un solo uso.
Bajo
Sin backups automáticos Solo existe un dump manual de marzo de alquileres_db.
